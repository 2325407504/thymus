Thymus
tar.gz .zip

thymus.js is a small (1k minified) JavaScript template library for including/substituting client side fragments using HTML5 resources that reside in different paths on a server and/or file system.

Internally, it dynamically loads the JQuery API (configurable) and uses JQuery $.get(...) calls to retrieve and insert fragments into pages. The process is similar to how JQuery $.load(...) works with page fragments with a few added benefits. thymus.js will automatically insert fragments into a page using HTML5 data attributes within the page and will update any relative src/href URLs within fragments to point to the proper path before insertion. It also supports nested fragments that may reside in other fragments. Click here for a full list of features.

This site actually uses thymus.js templates to resolve fragments for the top navigation bar, bottom navigation bar, example user profile drop down menu, page head and scripts. So, feel free to explore :)

Requirements
  • JQuery >= 1.9.1 (see JQuery URL attribute)

Features

  • Automatic fragment inclusion
  • Automatic fragment substitution
  • Automatic fragment src/href resolution for relative paths
  • Multiple fragment definitions in the same fragment resource
  • Relatively infinite nesting of fragments (fragments within fragments)
  • Head tag fragments
  • Fragment loaded event capture/execution
  • Great for prototyping web sites on the client!

Usage

To use thymus.js it obviously needs to be included in the page. Here is the minimal setup: 

<head>
<meta charset="utf-8" />
<!-- Thymus script -->
<script src="js/thymus.js" id="thymusScript" data-th-context-path="."></script>
</head>

There are two required thymus.js attributes. The thymusScript id is used to identify the thymus.js script in order to capture various attribute options on the script tag while the data-th-context-path is used for fragment path resolution and attribute path resolution. It should point to the root context of the application relative to the page in which thymus.js is loaded.

Fragments are simply defined using the data-th-fragment attribute and assigning a fragment unique value to it. Any other content that surrounds the fragment will be ignored. The site you are viewing uses the following fragment for the top navigation bar: 

<div data-th-fragment="topnav">
	...
</div>

Head Fragments

Almost every browser removes head attributes when parsing an HTML DOM. So, the thymus.js script definition needs to have an additional attribute to handle head fragment inclusions (subsititutions are treated the same as includes for head tags). If the browser doesn't strip out the data-th-include on the head tag it will be used. Otherwise, an attempt will be made to capture data-th-head-frag as in the following example: 

<head data-th-include="frags/head :: header">
<meta charset="utf-8" />
<!-- Thymus script -->
<script src="js/thymus.js" id="thymusScript"
	data-th-context-path="."
	data-th-head-frag="data-th-include=frags/head :: header"></script>
</head>

Although this duplicates the inclusion defintion it does provide for cross-browser support. If this redundency disrupts your inner chi you can ommit the data-th-include attribute from the head tag and the data-th-head-frag will still be picked up.

Script Fragments

Script tags will be inserted individually in the order they are defined within a fragment. Script tags found within a fragment will also fire a fragment load event for each script that is loaded within the fragment destination.

Fragment Load Events

Each time a fragment is loaded a fragment load event is fired. Also, each script tag within a fragment will also fire its own fragment load event. Event propigation is linear and non-cancellable and therefore, can only be defined on the thymus.js tag itself using the data-th-onfragloaded attribute. 

<head>
<meta charset="utf-8" />
<!-- Thymus script -->
<script src="js/thymus.js" id="thymusScript"
	data-th-context-path="." data-th-frag-extension=".htm"
	data-th-onfragloaded="init(event)"></script>
</head>

In the example above a function called init is defined as the data-th-onfragloaded. Every time a fragment is successfully loaded the an window.init function will be called. If the function contains an event parameter (w/o quotes) the fragment load event will be passed at the specified parameter index. The fragment load event contains the following properties:

Fragment Load Event

url
the URL of the fragment that was loaded
fragCount
the number of total fragments loaded
fragCurrTotal
the current number of total fragments that have or will be loaded (may grow as new script tags or nested fragments are discovered)
element
the DOM element that was included or subituted
error
the Error object (if the fragment loaded successfully, error will be null)

Load Complete Event

fragsNumLoaded
the total number of fragments loaded (including nested fragments and scripts)
errors
an Array of Error objects (if all fragments loaded successfully, errors will be null)

Path Resolution

Fragment Path Resolution

Because thymus.js runs on the client it's not inheriently application context aware. Therefore, thymus.js uses a configurable data-th-context-path attribute on the thymus.js script tag. It's value establishes the path to the root context of the application that will be resolved along with the fragment's inclusion URL. Fragment definitions can point to resources with different path levels. So, it's imporant that the data-th-context-path synchronizes with the path defined within the fragment. For example, a fragment that resides in example.com/my-app/frags/nav.htm may also be included in example.com/my-app/index.htm as well as example.com/my-app/user/info.htm. If the fragment is defined something like: 

<div data-th-fragment="topnav">
	...
</div>

And both example.com/my-app/index.htm and example.com/my-app/user/info.htm contain a corresponding include (or substitution): 

<div data-th-include="frags/nav/nav :: topnav">
	...
</div>

Then the thymus.js script definitions would be as follows: 

<head>
<meta charset="utf-8" />
<!-- example.com/my-app/index.htm -->
<script src="js/thymus.js" id="thymusScript" data-th-context-path="."
	data-th-frag-extension=".htm"></script>
</head>

<head>
<meta charset="utf-8" />
<!-- example.com/my-app/user/info.htm -->
<script src="../js/thymus.js" id="thymusScript"
	data-th-context-path="../" data-th-frag-extension=".htm"></script>
</head>

Notice that in both resources the data-th-context-path points to example.com/my-app with my-app being the root context (example.com/my-app/index.htm uses "." which infers the current path; which would be example.com/my-app/ while example.com/my-app/user/info.htm uses "../" which infers one directory up; which would be example.com/my-app/). This vasly simplifies fragment path resolution (escpecially for nested fragments) and ensures that all fragment inclusions/substitutions paths are relative to the root context path of the application.

Attribute Path Resolution

As with Fragment Path Resolution, relative URLs within fragments are resolved to their absolute URL counterparts relative to the root context of the application as defined by the data-th-context-path on script. Only href and src attributes with relative paths are eligable for updating. Any href/src attributes that contain URLs that begin with an IANA registered scheme or begin with a / will NOT be updated by thymus.js. When any href/src attributes within a fragment begin with a protocol-relative URL (i.e. //) and the page is being called using a file URI scheme the URL will default to http: in order to preserve external URL integrity. Here are a few examples: 

<!-- a page at: http://example.com/my-app/login.htm -->
<!-- with a data-th-context-path="." that contains a fragment with a -->
<a href="index.htm">...</a>
<!-- will be result in -->
<a href="http://example.com/my-app/index.htm"></a>

<!-- a page at: http://example.com/my-app/user/info.htm -->
<!-- with a data-th-context-path="../" that contains a fragment with a -->
<a href="index.htm">...</a>
<!-- will be result in -->
<a href="http://example.com/my-app/index.htm"></a>

<!-- a page at: http://example.com/my-app/index.htm -->
<!-- with a data-th-context-path="." that contains a fragment with a -->
<a href="user/info.htm">...</a>
<!-- will be result in -->
<a href="http://example.com/my-app/user/info.htm"></a>

<!-- a page at: file://example.com/my-app/index.htm (notice the file: scheme) -->
<!-- with a data-th-context-path="." that contains a fragment with a -->
<link
	href="//code.jquery.com/ui/jquery-ui-git.css"
	rel="stylesheet" />
<!-- will be result in -->
<link
	href="http://code.jquery.com/ui/jquery-ui-git.css"
	rel="stylesheet" />

Thymeleaf Integration

Thymeleaf is a great server-side template engine for Java and is the inspiration behind Thymus. In fact, by default Thymus uses the same default HTML5 attributes for includes/substitutions and fragment definitions. 

<head th:substituteby="frags/head :: header">
<meta charset="utf-8" />
<!-- Thymus script -->
<script th:remove="all" src="js/thymus.js" id="thymusScript" 
	data-th-context-path="." data-th-frag-extension=".htm"
	data-th-head-frag="th:substituteby=frags/head :: header"
	data-th-onfragsloaded="init(this)"></script>
</head>
